# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

A Claude Code plugin (`github-lifecycle`) that provides 18 slash commands for full development lifecycle automation with GitHub and Gitflow branching. Distributed via a custom marketplace (`AndrianopoulosGeo/claude-marketplace`).

## Architecture

### Plugin Structure

```
.claude-plugin/plugin.json    — Plugin metadata, version, marketplace registry
commands/                     — Slash command definitions (each .md = one command)
commands/_shared/             — Reusable fragments sourced by multiple commands
templates/                    — Files copied into consuming projects by /init-project
evals/                        — Eval specs for testing command behavior and trigger accuracy
docs/plans/                   — Implementation plans for this plugin's own development
```

### How Commands Work

Each `commands/*.md` file is a Claude Code slash command. The frontmatter defines `allowed-tools` and `description`. The body is a step-by-step prompt that Claude follows when the command is invoked.

Commands are **not code** — they're structured prompts with bash snippets, decision logic, and references to other skills/plugins.

### Shared Fragments

`commands/_shared/` contains reusable patterns that commands reference (not import — they're read inline):
- `load-config.md` — `.env.claude` loading and GitHub auth (used by all lifecycle commands)
- `state-management.md` — `.state.md` read/write patterns and track→step→next mappings

### Command Dependency Chain

Commands delegate to each other and to external plugins:

| Command | Delegates To |
|---------|-------------|
| `/commit` | `commit-commands:commit` (external plugin) |
| `/develop`, `/quick-fix`, `/hotfix` | `/commit` for all commits |
| `/develop` | `superpowers:executing-plans`, `pr-review-toolkit:code-simplifier`, `pr-review-toolkit:review-pr` |
| `/quick-fix`, `/hotfix` | `superpowers:test-driven-development`, `superpowers:verification-before-completion` |
| `/feature` | `superpowers:brainstorming`, `superpowers:writing-plans` |
| `/next` | Reads `.state.md` and invokes the appropriate next command |
| `/setup-infra` | (standalone — cloud-agnostic stub) |
| `/setup-pipeline` | `/setup-infra` for `.env.infra`; generates GitHub Actions workflow stubs with approval gates configured via environment reviewers |
| `/feature`, `/develop`, `/quick-fix`, `/hotfix` | `commands/_shared/load-decisions.md` for ADR context |
| `/feature`, `/develop`, `/hotfix` | `/compress-decisions` after emitting an ADR |
| `/decision` | `/compress-decisions` after every operation |

### Project Setup (one-time)

`/setup-infra` generates a cloud-agnostic `.env.infra` template; `/setup-pipeline` generates GitHub Actions workflow stubs (`ci.yml`, `deploy-staging.yml`, `deploy-prod.yml`) with TODO markers for the user's cloud auth.

```
/init-project → /setup-infra → /setup-pipeline
```

After `/init-project`, the docs folder includes a `decisions/` subfolder seeded with:

- `0000-template.md` (canonical ADR template)
- `INDEX.md` (empty index, regenerated by `/compress-decisions`)
- `README.md` (human-readable explanation)

### Workflow Tracks

Three tracks flow through commands in sequence, tracked by `.state.md`:

```
Feature:   /feature → /develop → /staging → /release
Quick-fix: /quick-fix → /staging → /release
Hotfix:    /hotfix (direct to main, then backmerge to develop)
```

## Key Conventions

### Commit Rules (enforced by /commit)

- Conventional Commits format: `<type>(<scope>): <summary>`
- **No AI attribution** — never include `Co-Authored-By`, `Generated by`, Claude, Anthropic, or any AI reference in commit messages
- Auto-update context docs (`CLAUDE.md`, `docs/architecture.md`, `docs/TESTING.md`, `docs/`) when code changes warrant it

### Configuration

All commands read from `.env.claude` (never hardcoded values). Required fields: `GITHUB_OWNER`, `GITHUB_REPO`, `DEPLOY_TARGET`, `TECH_STACK`.

`/setup-infra` generates a cloud-agnostic `.env.infra` template; `/setup-pipeline` generates GitHub Actions workflow stubs (`ci.yml`, `deploy-staging.yml`, `deploy-prod.yml`) with TODO markers for the user's cloud auth.

### State Management

Lifecycle commands read/write `.state.md` to track workflow progress. Status values: `idle`, `in-progress`, `blocked`, `awaiting-review`, `ready-to-promote`.

### Architecture Decisions (ADRs)

ADRs live at `docs/decisions/<NNNN>-<slug>.md` inside the `/docs` folder.
They are visible on github.com via normal markdown rendering — no separate
publish step. Reads happen locally — no API calls — to keep Claude's
context cheap.

- **`docs/decisions/INDEX.md`** is the always-loaded compressed
  summary. `/feature`, `/develop`, `/quick-fix`, `/hotfix` all read it via
  `commands/_shared/load-decisions.md`.
- **Full ADR files** are loaded only on demand (when a decision is being
  challenged or quoted).
- **`/decision`** creates/lists/supersedes/deprecates ADRs manually.
- **`/compress-decisions`** regenerates the index after any ADR change.
- **Hybrid linking:** when `/feature` or `/develop` emits a new ADR, the
  Feature issue description gets a one-line link to the docs page (the
  docs folder is the source of truth, the issue is the pointer).

**Why a separate `INDEX.md`?** Context-window economics. 50 ADRs × 200
tokens (full) = 10,000 tokens. 50 ADRs × 30 tokens (one-line summary) =
1,500 tokens. The index keeps room in the window for code, docs, and
library context.

## GitHub primitive mapping

| Concept | GitHub primitive |
|---|---|
| Work-item hierarchy | Parent Issue + Sub-issues (`gh api` REST endpoint `/repos/.../issues/{n}/sub_issues`) |
| Work-item type | Labels `type:feature`, `type:hotfix`, `type:quickfix`, `type:task` |
| Workflow state | Labels `state:in-progress`, `state:blocked`, `state:awaiting-review`, `state:ready-to-promote` (idle = no `state:*` label) + `.state.md` (source of truth) |
| Pull requests | `gh pr create` against `develop` (features) or `main` (hotfix) |
| Releases | `gh release create` |
| Wiki | `/docs/` in main repo (markdown rendered on github.com) |
| ADRs | `docs/decisions/NNNN-slug.md` |

## Evals

Eval specs live in `evals/` as JSON files. Two types:
- **Trigger evals** (`trigger-evals.json`) — test that the right command is invoked for a given prompt
- **Execution evals** (`*-evals.json`) — test that a command produces expected output sections

Run evals via the `skill-creator` plugin: `/skill-creator run evals for <command> using evals/<file>.json`

## Version & Distribution

- Version is in `.claude-plugin/plugin.json` — bump on every release
- Source repo: GitHub (`origin`) — `AndrianopoulosGeo/github-lifecycle`
- Marketplace version is in a separate repo (`AndrianopoulosGeo/claude-marketplace`) in `.claude-plugin/marketplace.json` — must be bumped separately after pushing to GitHub
- **Both remotes must be pushed** for consumers to get updates
