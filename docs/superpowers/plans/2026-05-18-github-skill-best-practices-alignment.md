# GitHub Lifecycle — Skill Best-Practices Alignment Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Apply the same Anthropic Agent Skills alignment done on `azure-devops-lifecycle` to the `github-lifecycle` plugin — adapted to its actual state.

**Architecture:** Each command becomes a thin orchestrator referencing one-level-deep fragments. Tech-stack specifics move to per-stack fragments. The two oversized commands are decomposed under 500 lines.

**Tech Stack:** Markdown command files, YAML frontmatter, Bash, `gh` CLI. Verification via `grep`/`wc` and the `evals/` suites.

## Background

A review of `github-lifecycle` against Anthropic's Agent Skills guidance found it in **better shape** than `azure-devops-lifecycle` was. The findings that apply:

- **Finding 1 — descriptions:** 5/18 commands have a `description`; **13 are missing**.
- **Finding 4 — file length:** `develop.md` (972 lines) and `feature.md` (540) exceed the 500-line guideline. `setup-pipeline.md` is only 107 lines — fine.
- **Finding 5 — hardcoded stack:** `feature.md` and `develop.md` hardcode a Next.js stack in their plan templates.
- **Finding 6 — repeated build block:** the `nextjs/dotnet/python` build block repeats ~6× in `develop.md`.
- **Finding 8 — description quality:** `commit.md`'s description states what but not when.
- **Expert Voice:** 5 non-deprecated commands lack the `> **Expert Voice:**` header.

**Findings that do NOT apply** (and why): Windows Docker path — `develop.md` Phase 3.0 is already an OS-agnostic warning. Dead fragments — all 5 `_shared/` fragments are already wired (`load-config.md` used by 10 commands). Pipeline extraction — `setup-pipeline.md` is 107 lines with no embedded YAML.

**GitHub-specific adaptations vs the azure plan:** ADR path is `docs/decisions/` (not `docs/wiki/decisions/`); `feature_id` is a GitHub Issue number; branch base is `main`; `publish-wiki` is a deprecated no-op (skip its Expert Voice).

**Convention:** Conventional Commits, NEVER any AI attribution (`Co-Authored-By`, `Generated by`, Claude, Anthropic, AI).

## File Structure

**New `commands/_shared/` fragments:**
- `stack-nextjs.md`, `stack-dotnet.md`, `stack-python.md`, `plan-template.md` — copied verbatim from the sibling `azure-devops-lifecycle` plugin (platform-agnostic).

**Modified:** all 18 `commands/*.md` get/keep a `description`; `develop.md` + `feature.md` decomposed; `CLAUDE.md`; `.claude-plugin/plugin.json`.

>>>>>> BEGIN FILE
## Shared: Emit an Architecture Decision Record (ADR)

> Used by: `/feature` Step 5.5, `/develop` Phase 9. Read inline when the work
> introduces or changes an architectural decision. Most work does NOT — it
> implements existing patterns.

### Trigger conditions (any one is enough)

- A choice was made between 2+ approaches at the architectural level
  (e.g. client vs. server component as a class decision, SignalR vs. polling,
  JWT vs. session cookies, raw SQL vs. ORM).
- A new library, service, or external dependency is introduced.
- The way one layer interacts with another changes.
- The work deliberately departs from an existing pattern (not a one-off).

If no trigger fires, skip ADR emission entirely.

### Emit procedure

1. **Precondition** — verify the decisions folder is scaffolded:

   ```bash
   if [ ! -d docs/decisions ] || [ ! -f docs/decisions/0000-template.md ]; then
     echo "ERROR: docs/decisions/ not scaffolded. Run /init-project first."
     exit 1
   fi
   ```

2. Determine the next ADR number:

   ```bash
   LAST=$(ls docs/decisions/ 2>/dev/null | grep -E '^[0-9]{4}-' | grep -v '^0000-' | sort | tail -1 | cut -d'-' -f1)
   NEXT=$(printf "%04d" $((10#${LAST:-0} + 1)))
   SLUG="<kebab-case slug from the decision title>"
   ADR_PATH="docs/decisions/${NEXT}-${SLUG}.md"
   ```

3. Copy the template and fill it in:

   ```bash
   cp docs/decisions/0000-template.md "$ADR_PATH"
   ```

   Edit `$ADR_PATH` with: the title (`NNNN — present-tense imperative`),
   `status: accepted`, `date: <today>`, `feature_id: <PARENT>` (the GitHub
   Issue number), `tags` (relevant area), and the four canonical sections
   (Context, Decision, Alternatives, Consequences).

4. If this ADR supersedes another, set `supersedes: NNNN` in the new ADR's
   frontmatter, and update the old ADR's `superseded_by: NNNN` and
   `status: superseded`.

5. Regenerate the index by invoking the `Skill` tool with
   `skill: "compress-decisions"`.

6. Add a one-line ADR link to the parent GitHub Issue body or a comment (the
   decision record is the source of truth; the issue is the pointer):

   ```markdown
   ## Architecture Decision

   **ADR-NNNN — <title>**
   Status: accepted (<date>)
   Decision: <one-sentence summary>

   Full record: docs/decisions/NNNN-<slug>.md
   ```
<<<<<< END FILE
